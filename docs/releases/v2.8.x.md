# Release Notes — v2.8.x

> Draft — not yet tagged. All changes are merged to `main`.

---

## Summary

This release completes the stabilization cycle that began with the move to a `DeviceClient`-owned
IO architecture. No runtime Modbus behavior has changed. All entity IDs, register addresses,
service IDs, translation keys, and unique IDs remain unchanged from v2.7.x.

---

## Changes

### Fan percentage fix (#1682)

**Problem:** The fan percentage property on `fan.thesslagreen_ventilation` could return values
above 100 (e.g. 109) when the device reported a raw airflow value higher than the rated maximum.
Home Assistant requires fan percentage to be in the range 0–100.

**Fix:** The `percentage` property in `fan.py` now clamps the returned value to 100. The raw
value is preserved in the `supply_percentage` attribute for diagnostics.

**Evidence:** Unit test `test_fan_percentage_109_clamped_to_100` passes.

---

### Dangerous/advanced entities moved to `entity_category=config` (#1683)

**Problem:** Entities that write to dangerous or advanced device registers (e.g. temperature
limits, motor control, bypass overrides) were visible by default in the HA frontend alongside
normal operational entities.

**Fix:** All dangerous and advanced entities now carry `entity_category=config`. They remain
**enabled by default** — `entity_registry_enabled_default` is not set to `False`. The change
only affects where they appear in the HA UI (under the Device configuration section).

**No behavior change:** No entity IDs, unique IDs, or register mappings changed.

---

### DeviceClient IO ownership cleanup (#1684)

**Problem:** After earlier refactor slices, 5 coordinator delegate methods still forwarded
low-level Modbus IO calls through the coordinator rather than through `DeviceClient` directly.
This meant `DeviceClient` was not yet the true single owner of device-domain IO.

**Fix:**
- Removed 5 delegate methods from `coordinator.py`:
  `_get_client_method`, `_mark_registers_failed`, `_clear_register_failure`,
  `_find_register_name`, `_process_register_value`.
- Removed `_ModbusIOMixin` from `ThesslaGreenModbusCoordinator` inheritance.
- `coordinator/update.py` now calls `coordinator.device_client._read_all_register_data()`
  instead of going through the coordinator.
- `coordinator/schedule.py` write-path call sites updated to use `self._device_client.*`.
- The failed-register tracking optimization (`_failed_registers`) now works correctly because
  the owner passed to `read_batches.py` and `read_bits.py` is `device_client` (which holds the
  real set), not the coordinator (which had a property proxy removed in slice-1).

**New test coverage:** `tests/test_coordinator_io_ownership.py` — 15 tests.

### CI / Python 3.13 validation (#1686)

- Ruff import-order and format checks are now **blocking** in the lint CI job.
  Previously they ran in a non-blocking `ruff-adoption-signal` job.
- `compare_airpack4_vendor_coverage.py`, `check_translations.py`, and
  `compare_registers_with_reference.py --show-renames` added to the lint job.
- `pytest --collect-only -q` added as a separate step before the full test run.
- `ruff-adoption-signal` job removed (all checks are now in lint/tests).
- `docs/development_validation.md` added with Python 3.13 environment setup instructions.

### pymodbus 4.0 migration plan added (#1685)

`docs/pymodbus_4_migration_plan.md` documents the current pymodbus 3.x API surface, all
known risk areas for a 4.0 migration, a 16-scenario test matrix, and the step-by-step
migration strategy.

**No code change.** The `pymodbus>=3.6.0,<4.0` pin in `manifest.json` is unchanged.

---

### Core consolidation plan added (#1685)

`docs/core_consolidation_plan.md` inventories all 25 `core/` files, proposes a target
layout with `connection/` and `read/` sub-packages, defines 4 safe consolidation slices,
and sets prerequisites (real-device validation must come first).

**No code change.** All consolidation is deferred until real-device validation is complete.

## Known limitations / remaining work

1. **Real-device validation:** Formal evidence (logs, screenshots, entity counts) has not
   yet been committed. The integration is user-reported working on AirPack 4 but formal
   sign-off requires evidence for all items in `docs/real_device_validation.md` section 3.

2. **Core consolidation:** Deferred until real-device validation is complete (see
   `docs/core_consolidation_plan.md`).

3. **pymodbus 4.0 migration:** Planned but not started. Will be done on a separate branch
   after real-device validation (see `docs/pymodbus_4_migration_plan.md`).

4. **quality_scale upgrade from bronze to silver:** Requires real-device evidence for all
   required items. Not changed in this release.
